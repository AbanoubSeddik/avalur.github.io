Python

										  
def fibonacci(n):
	curr, prev = 1, 0
	for i in range(n):
		curr, prev = curr+prev, curr
	return curr

C++

										  
int fibonacci(int n){
	int curr = 1, prev = 0;
	for(int i = 0; i < n; i++){
		int temp = curr;
		curr = curr + prev;
		prev = temp;
	}
	return curr;
}

Haskell

										  
fibonacci :: Integer -> Integer
fibonacci n = fibs !! (fromIntegral n)
  where
	fibs = 1 : 1 : zipWith (+) fibs (tail fibs)

Kotlin

										  
fun fibonacci(n: Int): Int {
	var curr = 1
	var prev = 0
	for (i in 0 until n) {
		val temp = curr
		curr += prev
		prev = temp
	}
	return curr
}

Rust

										  
fn fibonacci(n: u32) -> u32 {
	let mut curr = 1;
	let mut prev = 0;
	for _ in 0..n {
		let temp = curr;
		curr = curr + prev;
		prev = temp;
	}
	curr
}

Compilation and Interpretation

Compilation

  • Source code $\to$ Machine code
  • Involves several steps like lexing, parsing, semantic analysis, optimization, code generation, and linking
  • Compiled languages include C++, Go, Rust, and Kotlin

C++ and Rust use ahead-of-time compilation. Here is how it works:

  • *.cpp $\to$ *.obj
  • *.obj are machine code, but it may also include placeholders for other files
  • These placeholders are filled in by the “linker”, producing an executable

Go also compiles directly to machine code

  • but it compiles very quickly
  • and produces a single binary without external dependencies
  • which makes deploying Go applications very convenient

Kotlin is a bit of an exception here because it is both compiled and interpreted

  • Kotlin can be compiled to Java bytecode to run on the JVM, and it can also be transcompiled into JavaScript
  • The primary advantage of compiled languages is their performance
  • There's a downside that programs have to be recompiled every time the source code changes

Interpretation

  • Interpreted languages (like Python and JavaScript) don't need a compilation step
  • Interpreter reads and executes the source code line by line
  • JavaScript was originally an interpreted language (hence the 'script' in the name) aimed for web browsers
  • Modern JavaScript is often compiled Just-In-Time (JIT), which gives an efficient compromise between interpreter and compiler
  • With the time the gap between interpreted and compiled languages is closing

Virtual Machines

  • VM can be thought of as a 'computer within a computer' that provides the functionality of a physical computer and allows running multiple operating systems or environments isolated from each other
  • A great example is the Java Virtual Machine (JVM). When you compile Java code using the Java Compiler (javac), it doesn't create machine code — it creates bytecode. JVM is a part of the Java Runtime Environment (JRE), which interprets this bytecode and runs the Java program
  • This provides Java with its infamous "Write Once, Run Anywhere" capability and makes Java highly portable

Concurrency vs. Parallelism

  • Concurrency refers to the ability of a program to handle multiple tasks at the same time, but not necessarily simultaneously
  • Parallelism, on the other hand, refers to the execution of multiple tasks or processes simultaneously — splitting a program into smaller sub-tasks that can be processed in parallel
  • So, a concurrent system could be multitasking on a single core by dealing with multiple tasks, rapidly switching between them, while a parallel system would use multiple cores to execute multiple tasks at the same time

Language Approaches

  • Java uses Threads and has rich built-in support for multithreading programming
  • In C, parallelism can be achieved through multiple processes using fork system call, and multiple threads using library like POSIX threads (pthreads)
  • Historically JavaScript was single-threaded, but with HTML5 Web Workers API, JavaScript code can run in a background thread separate from the main execution thread
  • Python has built-in support for thread-based parallelism via its threading library, and process-based parallelism via its multiprocessing library. However, due to the Global Interpreter Lock (GIL), the threading library doesn't allow true parallelism in Python
  • Go has built-in support for parallelism via goroutines (lightweight threads managed by the Go runtime) and channels for communication between those goroutines
  • Kotlin allows to launch coroutines that are very light-weighted threads
  • Rust supports concurrency through a combination of OS threads and lighter weight "green" threads or tasks. In Rust, sharing data between threads is carefully controlled, which can help prevent concurrency issues such as race conditions
